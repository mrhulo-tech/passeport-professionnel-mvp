# Backlog MVP initial — Formation professionnelle / employabilité / IA literacy

## 1. Objectif du backlog

Ce backlog vise à structurer un MVP démontrable, simple et crédible. Il priorise les éléments nécessaires pour produire rapidement une preuve de valeur.

## 2. Principes de priorisation

Toujours prioriser :
- visibilité de la valeur ;
- simplicité technique ;
- lisibilité produit ;
- robustesse minimale ;
- réutilisation de l’architecture existante quand possible.

Éviter :
- modèles complexes trop tôt ;
- logique opaque ;
- intégrations lourdes non essentielles au MVP.

## 3. Backlog priorisé

### EPIC 1 — Profil dynamique de préparation professionnelle

#### Story 1.1
En tant qu’utilisateur, je veux voir un profil synthétique afin de comprendre rapidement ma situation actuelle.

**Critères d’acceptation**
- nom du profil ;
- niveau ou statut de progression ;
- objectif de parcours ou cible professionnelle ;
- vue lisible du profil.

#### Story 1.2
En tant qu’équipe pédagogique, je veux associer un profil à des compétences observables afin de guider l’accompagnement.

**Critères d’acceptation**
- liste de compétences visibles ;
- formulation compréhensible ;
- lecture non déterministe.

---

### EPIC 2 — Données observables et indicateurs

#### Story 2.1
En tant que système, je veux agréger des données simples du parcours afin de produire des indicateurs utiles.

**Données minimales**
- progression ;
- complétion ;
- participation ;
- résultats à tâches ;
- régularité.

#### Story 2.2
En tant qu’utilisateur, je veux voir des indicateurs interprétables afin de comprendre ma progression.

**Critères d’acceptation**
- 3 à 5 indicateurs maximum ;
- libellés clairs ;
- lien explicite avec les données observables.

---

### EPIC 3 — Recommandations personnalisées

#### Story 3.1
En tant qu’utilisateur, je veux recevoir des recommandations de prochaines étapes afin de poursuivre mon parcours.

**Critères d’acceptation**
- 2 à 3 recommandations maximum ;
- logique explicable ;
- lien avec les indicateurs.

#### Story 3.2
En tant qu’équipe pédagogique, je veux comprendre pourquoi une recommandation est affichée afin de pouvoir l’expliquer.

**Critères d’acceptation**
- logique simple ;
- relation visible entre données, indicateurs et recommandation.

---

### EPIC 4 — Évaluation par tâches

#### Story 4.1
En tant que concepteur pédagogique, je veux utiliser des tâches simples plutôt que des tests abstraits afin de rester aligné sur les compétences réelles.

**Critères d’acceptation**
- tâches ou simulations courtes ;
- résultats exploitables dans le profil ;
- pas de psychométrie lourde.

#### Story 4.2
En tant qu’apprenant, je veux que mes tâches contribuent à une lecture de progression afin que le tableau de bord ait du sens.

**Critères d’acceptation**
- contribution visible au profil ;
- feedback simple ;
- cohérence avec les recommandations.

---

### EPIC 5 — Dashboard MVP

#### Story 5.1
En tant qu’utilisateur ou partenaire, je veux un dashboard clair afin de comprendre le produit rapidement.

**Critères d’acceptation**
- profil ;
- métriques ;
- indicateurs ;
- recommandations ;
- note de cadrage éthique.

#### Story 5.2
En tant que démonstrateur, je veux montrer plusieurs profils utilisateurs afin d’illustrer la personnalisation.

**Critères d’acceptation**
- au moins 2 profils distincts ;
- variations visibles ;
- navigation simple.

---

### EPIC 6 — Intégration LMS minimale

#### Story 6.1
En tant que système, je veux consommer des données depuis le LMS ou une structure simulée afin de démontrer l’intégration.

**Critères d’acceptation**
- modèle de données clair ;
- backend fonctionnel ;
- données mockées ou réelles limitées.

#### Story 6.2
En tant qu’équipe projet, je veux préserver une architecture simple afin de faciliter le build.

**Critères d’acceptation**
- frontend lisible ;
- backend stable ;
- pas de dépendance inutile.

## 4. Priorités immédiates

### Priorité P0
- profil dynamique ;
- indicateurs interprétables ;
- recommandations simples ;
- dashboard démontrable ;
- 2 profils utilisateurs ;
- logique explicite.

### Priorité P1
- tâches mieux structurées ;
- IA literacy ;
- cybersécurité ;
- enrichissement des sources de données.

### Priorité P2
- analytics avancés ;
- intégration LMS plus profonde ;
- parcours multi-secteurs ;
- reporting partenaire enrichi.

## 5. Conclusion

Ce backlog est conçu pour livrer un MVP crédible en réutilisant les apprentissages du projet initial tout en l’orientant vers la formation professionnelle et l’employabilité.
